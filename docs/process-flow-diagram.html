Passenger Journey Flow (Web)

End-to-end flow from landing page to viewing live bus status with time windows. The primary user-facing workflow of the platform.

App.jsx TrackingDashboard.jsx RouteDetails.jsx routes.py query.py
1. User Opens Website
Landing page renders Hero + TrackingDashboard components
App.jsx initializes with currentPage = 'track'. The <Hero> component renders the navigation header with district selector, language picker (English/Tamil/Hindi), and dark mode toggle. <TrackingDashboard> loads below as the main content area. On mount, loadActiveBuses() auto-fetches currently running buses.

GET /api/v1/routes/active-buses
2. Select District & Boarding Point
User picks district from dropdown, stops populate dynamically
District selection is stored in localStorage('userDistrict'). Available districts include Chennai, Salem, Coimbatore, Ernakulam, Thiruvananthapuram, and dynamic fallback for any other district. Stop lists come from districtStops.js data file. When district changes, previous search state is cleared via sessionStorage.
3. Select Destination & Search
Form validation: boarding !== destination, both required
The search form uses handleSearch() which calls the API with boarding point, destination, and active district.

GET /api/v1/routes/search?from_stop=X&to_stop=Y&district=Z

Three possible outcomes: results (matching routes found), empty (boarding = destination), or unavailable (no routes for this pair).
4. System Searches Routes
Backend queries Route + Trip tables; frontend generates district routes
The backend routes.py searches the database for active routes matching the from/to stops. The frontend also has generateDistrictRoutes() which creates realistic route data for each district. Each route object contains: id, route, status, currentLocation, expectedArrival, confidence, delayReason, and district.
5. Display Matching Routes with Live Status
Grid of route cards showing real-time status indicators
Routes render in a responsive 4-column grid. Each card shows:
• Route ID (e.g., 500A-1X)
• Status badge: ON TIME (green) or DELAYED (red)
• RUNNING TODAY confirmation badge
• Current Location of the bus
• ETA window (e.g., "5 - 10 Minutes")

Status types per TripStatusEnum:
6. Click Route Card for Details
navigateTo('routeDetails') with full route data payload
handleRouteClick(routeData) saves the route object into globalRouteDetails state and navigates to the <RouteDetails> component. This page shows: full stop sequence, timing information, bus type, and live GPS status. From here, passengers can access Emergency Dashboard or submit Feedback.
7. Optional: Emergency / Feedback
EmergencyDashboard.jsx or ServiceReporting.jsx
From route details, passengers can:
• Emergency tab → Access safety escalation panel (see Safety Escalation Flow)
• Service Reporting → Submit feedback via /api/v1/feedback
• Notify ETA → Get SMS notification via /api/v1/notify/eta using AWS SNS

Trip Status Types (TripStatusEnum)

Status Enum Value Visual Indicator Meaning
Running Today RUNNING Green badge ETM login confirmed; bus is active on route
Not Running NOT_RUNNING Red badge Admin-cancelled; bus will not operate today
Status Not Confirmed UNCONFIRMED Yellow badge No ETM login and past timeout (30 min)
Service Discontinued DISCONTINUED Gray badge Route permanently removed from service

Operator Authentication & Dashboard Flow

Secure OTP-based authentication via AWS SNS, followed by route management and trip activation/deactivation.

Login.jsx OTP.jsx OperatorDashboard.jsx auth.py operator.py AWS SNS

Phase 1: Authentication

1. Operator Opens Login Page
Login.jsx renders mobile number input + CAPTCHA
App.jsx routes to <Login> component when currentPage = 'login'. The login form requires a valid 10-digit Indian mobile number. CAPTCHA verification is performed client-side before the form can be submitted.
2. Submit Mobile Number
POST /api/v1/auth/send-otp with phone_number
POST /api/v1/auth/send-otp

Backend validates: phone.isdigit() and len(phone) == 10. Generates a 6-digit OTP via random.randint(100000, 999999). Phone number is hashed using SHA-256 (first 16 chars) for privacy: hashlib.sha256(phone.encode()).hexdigest()[:16]. OTP stored in database OTPRecord table with 5-minute expiry (OTP_EXPIRY_SECONDS = 300).
3. AWS SNS Sends OTP via SMS
Transactional SMS to +91XXXXXXXXXX with SenderID "TRANSIT"
boto3.client("sns") publishes the message:
"Transit India: Your OTP is XXXXXX. Valid for 5 minutes. Do not share this code."

SMS attributes: SenderID = "TRANSIT", SMSType = "Transactional". Region: ap-south-1 (Mumbai). If SNS fails, OTP is still returned in debug_otp field for development.
4. Verify OTP
POST /api/v1/auth/verify-otp validates OTP and assigns role
POST /api/v1/auth/verify-otp

Validation checks: OTP must be exactly 6 digits, must match stored OTP, must not be expired. After verification:
• OTP record is deleted (one-time use)
• System checks Operator table for matching phone_hash
• Returns role: "operator" with operator_id if found, else role: "passenger"
• Auth token generated: sha256(phone + timestamp)[:32]
5. Redirect to OperatorDashboard
navigateTo('operatorDashboard') with operator_id stored in session
If role === "operator", the system navigates to the OperatorDashboard. The operator_id (e.g., "OP-001") is stored in localStorage for subsequent API calls. If role is "passenger", user is redirected to the TrackingDashboard instead.

Phase 2: Trip Management

6. Browse & Select Route
GET /api/v1/operator/routes lists all active routes
GET /api/v1/operator/routes

Returns all routes where Route.status == "active". Each route includes: id, name, stops_sequence (ordered list), and stop_count. Operator selects a route to view its GPS-enriched stop details.
7. View Route Stops with GPS
GET /api/v1/operator/routes/{route_id} enriched with coordinates
GET /api/v1/operator/routes/{route_id}

Each stop in stops_sequence is enriched by querying the Stop table for matching name_en. Returns: name, latitude, longitude for map display.
8. Enter Bus Number & Activate Trip
POST /api/v1/operator/trips/activate creates Running Today trip
POST /api/v1/operator/trips/activate

Payload: { route_id, bus_number, operator_id }. Backend verifies route exists and operator is valid. Creates Trip record with: id = "T-{uuid[:8]}", status = "Running Today", etm_login_time = now. Bus number is normalized: strip().upper().
9. Monitor Active Trips
GET /api/v1/operator/trips/active?operator_id=OP-001
GET /api/v1/operator/trips/active?operator_id={id}

Filters trips where status == "Running Today", administrative_note contains operator_id, and etm_login_time is today. Returns list of active trips with route names for the dashboard view.
10. Deactivate Trip on Completion
POST /api/v1/operator/trips/{trip_id}/deactivate sets status to "Completed"
POST /api/v1/operator/trips/{trip_id}/deactivate

Updates Trip.status = "Completed" and Trip.last_updated = utcnow(). The trip no longer appears in active bus lists or passenger search results.
Auth Data Flow: Phone number is never stored in plaintext. All records use phone_hash = SHA-256[:16]. OTP records are deleted after successful verification. Operator records are pre-seeded in the database with known phone hashes.

Safety Escalation Flow

Emergency incident reporting system with control room integration, rate limiting, and GPS-verified submissions.

EmergencyDashboard.jsx safety.py rate_limiter.py SafetyEscalationRecord
1. Passenger Accesses Emergency Tab
navigateTo('emergency') from route details or navigation
The <EmergencyDashboard> component receives routeData as a prop containing the active route/bus information. The page displays the current transit session info (route number, bus ID) with a "LIVE GPS VERIFIED" indicator, followed by the escalation panel and emergency helpline numbers (112, 108, 1091).
2. Select Incident Category
Four categories defined by IncidentCategoryEnum
Categories from schemas.py / IncidentCategoryEnum:
• Harassment - Covers harassment and staff misconduct
• Threat - Threat to life situations
• Fight - Physical altercation on bus
• Medical - Medical emergency

Frontend maps "Staff Misconduct" to the "Harassment" backend enum value via categoryMap.
3. Submit Escalation Report
POST /api/v1/safety/escalate with phone_hash, bus_number, category
POST /api/v1/safety/escalate

Payload: { phone_number, bus_number, incident_category, trip_id }

Backend generates unique escalation ID: "esc-{uuid[:8]}". Phone is hashed for privacy. Creates SafetyEscalationRecord in database with control_room_status = "Pending".
4. Rate Limit Check
Max 3 escalations per day per phone number
RateLimiter enforces: LIMITS["safety"] = { max: 3, ttl: 86400 } (3 per 24 hours).

Redis key pattern: rate:{phone_hash}:safety. If limit exceeded, returns (False, 0) and the request is rejected. This prevents abuse of the emergency system.
5. Control Room Receives Alert
EscalationStatusEnum: Pending -> In Progress -> Resolved
The escalation record is created with status "Pending". Control room operators review and update the status through the admin interface. Status progression follows EscalationStatusEnum:
• Pending → Initial state after submission
• In Progress → Control room is investigating
• Resolved → Issue addressed, resolution notes added
6. Confirmation Modal Displayed
Success modal with escalation ID and GPS forwarding confirmation
The frontend shows a modal: "Escalation Successfully Logged" with the message: "Your details and live GPS coordinates have been securely forwarded to the Depot Control Room. A safety officer is reviewing this case and will contact the bus crew immediately."

Additionally, direct dial buttons are available: 112 (Police), 108 (Ambulance), 1091 (Women Safety) with a 5-second countdown before auto-dialing.
Legal Notice (displayed in UI): "Misuse of these emergency services results in severe criminal penalties under suitable sections of the law. All escalations are logged with verified GPS and device credentials."

SMS Query Flow

SMS-based bus status inquiry for feature phone users. Parses natural language messages and returns arrival information within 10 seconds.

sms.py Trip table AWS SNS (Inbound)
User SMS Feature Phone
→
SMS Gateway AWS SNS Webhook
→
Parse Message Regex Pattern
→
Query Database Trip Lookup
→
SMS Response < 10 seconds
1. User Sends SMS
Format: "BUS <BusNumber> <StopName>" (e.g., "BUS 42B Adyar")
The SMS webhook receives messages at /api/v1/sms/webhook. Expected format matches the regex pattern:
SMS_PATTERN = r'^bus\s+(\S+)(?:\s+(.+))?$' (case-insensitive)

Group 1: Bus number (required, uppercased). Group 2: Stop name (optional, title-cased).
2. Parse SMS Message
parse_sms() extracts bus_number and optional stop_name
parse_sms(message) returns:
• None if message is empty or does not match pattern
• { "bus_number": "42B", "stop_name": "Adyar" } on success

Invalid format returns: "Invalid format. Please use: BUS <BusNumber> <StopName>. Example: BUS 45A Adyar"
3. Query Trip Database
Lookup latest trip by bus_number, ordered by scheduled_departure DESC
Database query: session.query(Trip).filter(Trip.bus_number == bus_number).order_by(Trip.scheduled_departure.desc()).first()

If trip found: status from DB, arrival based on ETM login.
If no trip found: defaults to status = "Running Today", arrival = "10-15 minutes".
4. Calculate Arrival Window
ETM login present: 5-12 min, absent: 15-25 min
Arrival calculation in SMS handler:
• ETM login exists (trip.etm_login_time is not None): "5-12 minutes" (higher confidence)
• No ETM login: "15-25 minutes" (lower confidence, wider window)
• No trip found: "10-15 minutes" (fallback default)
5. Send Response SMS
Response within 10 seconds: "Bus 42B: Running Today. Arrival: 5-12 minutes."
Response format: "Bus {bus_number}: {status}. Arrival: {arrival}. Time: {HH:MM}"

Example: "Bus 42B: Running Today. Arrival: 5-12 minutes. Time: 14:30"

Rate limit: LIMITS["sms"] = { max: 10, ttl: 3600 } — 10 queries per hour per phone number.
Rate Limiting: The RateLimiter class uses Redis with key pattern rate:{phone}:sms. Max 10 queries/hour for SMS, 5 queries/hour for IVR. TTL is enforced via Redis EXPIRE command.

Arrival Prediction Flow (Backend)

Multi-factor confidence-based arrival prediction engine. Produces time windows (never exact ETAs) based on ETM, GPS, and historical data sources.

arrival_prediction.py trip_status.py ml/confidence.py ml/forecasting.py ArrivalPrediction schema
Query Request trip_id + stop_id
→
TripStatusService check_status()
→
ConfidenceCalculator 0.0 - 1.0 score
→
Window Calculation confidence_to_window()
→
Time Window "7-14 minutes"
1. Prediction Query Received
ArrivalPredictionService.predict() called with trip context
Input parameters to predict():
• trip_id - Unique trip identifier
• stop_id - Target stop for arrival estimate
• data_recency_minutes - How fresh the latest data is
• has_gps - Boolean, whether GPS data is available
• historical_accuracy - Past prediction accuracy (0.0-1.0)
• ml_predicted_minutes - ML model's point estimate (optional)
• ml_variance - ML model's prediction variance (optional)
2. Check Trip Status via TripStatusService
Cache check (Redis) then DB fallback; TIMEOUT_MINUTES = 30
TripStatusService.check_status() logic:
• First check Redis cache: trip:{trip_id} key
• If cached, return TripStatusEnum(cached["status"])
• If ETM login time exists → RUNNING
• If past scheduled_departure + 30 min with no ETM → UNCONFIRMED

Cache TTL: CACHE_TTL_SECONDS = 86400 (24 hours).
3. ML Prediction Check
If ml_predicted_minutes is None, return "Status Not Confirmed"
If the ML model has no prediction (ml_predicted_minutes is None), the service immediately returns:
{ "arrival_window": "Status Not Confirmed", "confidence": 0.0, "prediction": None }

This ensures the system never fabricates arrival estimates when data is insufficient.
4. Calculate Confidence Score
ConfidenceCalculator weighs data recency, GPS, history, variance
ConfidenceCalculator.calculate() factors:
• Data Recency - Fresher data = higher confidence
• GPS Availability - GPS present = significant confidence boost
• Historical Accuracy - Past accuracy of this route/time
• Prediction Variance - Lower ML variance = higher confidence

Default variance if not provided: ml_variance or 15.0
5. Convert Confidence to Time Window
confidence_to_window() maps score to min/max arrival range
confidence_to_window(confidence, ml_predicted_minutes) determines the window width:
• High confidence (>0.8) → Narrow window (~5 min spread)
• Medium confidence (0.5-0.8) → Moderate window (~10 min spread)
• Low confidence (0.2-0.5) → Wide window (~15 min spread)
• Very low (<0.2) → Returns None ("Status Not Confirmed")

Returns { "min": X, "max": Y } or None.
6. Return ArrivalPrediction Object
Structured response with window, confidence, and data sources
Final ArrivalPrediction schema fields:
• prediction_id: "pred-{uuid[:8]}"
• predicted_arrival_min / predicted_arrival_max: integer minutes
• confidence_level: float 0.0 - 1.0 (validated by Pydantic Field(ge=0.0, le=1.0))
• data_sources_used: list, always includes ["ETM", "Historical"], adds ["GPS"] if available
• query_time: timestamp of prediction

Output: { "arrival_window": "7-14 minutes", "confidence": 0.72, "prediction": {...} }

Confidence-to-Window Mapping

High Confidence
> 0.8
~5 min window
Medium Confidence
0.5 - 0.8
~10 min window

~15 min window

< 0.2

Design Principle: The system intentionally provides time windows (e.g., "7-14 minutes") rather than exact ETAs. This sets realistic expectations and accounts for inherent uncertainty in public transit timing. The window width directly reflects data quality.

Data Seeding / Admin Flow

Database initialization workflow that populates realistic transit data including routes, stops, trips, and operator records.

admin.py data/seed_db.py data/seed.py SQLite / PostgreSQL
1. Admin Triggers Seed Endpoint
POST /api/v1/admin/seed (or via frontend api.seedDatabase())
POST /api/v1/admin/seed

Triggered from the frontend via api.seedDatabase() or directly via the API. The endpoint first ensures all database tables exist via Base.metadata.create_all(engine), then clears existing Trip, Route, and Stop data before re-seeding.
2. Generate Routes
generate_routes(25) creates routes across districts
generate_routes(25) creates 25 routes (production seed_db uses 50). Each route has:
• id: unique route identifier
• name: human-readable name
• stops_sequence: ordered list of stop names
• status: "active"

Routes span across multiple districts covering 5 bus types.
3. Generate Stops with GPS Coordinates
generate_stops() creates 95+ stops with multilingual names
Each stop record includes:
• id: unique stop identifier
• name_en: English name
• name_ta: Tamil name
• name_hi: Hindi name
• latitude / longitude: GPS coordinates

These multilingual names support the platform's language switcher (English, Tamil, Hindi).
4. Generate Trip Data (7 Days)
generate_trips(routes, days=7) with realistic ETM login patterns
generate_trips(routes_data, days=7) creates trip records spanning 7 days. Each trip includes:
• id: "T-{uuid[:8]}" format
• route_id: reference to a generated route
• bus_number: realistic bus plate numbers
• scheduled_departure: ISO datetime
• status: mix of Running Today / Not Running / etc.
• etm_login_time: simulated ETM device login times (null for some to test "Status Not Confirmed" logic)
5. Seed Operator Records
Pre-registered operators with hashed phone numbers
Operator seed data from admin.py:
OPERATOR_PHONES = [
  { "phone": "97867XXXXX", "name": "Operator 1" },
  { "phone": "97150XXXXX", "name": "Operator 2" }
]

Phone numbers are SHA-256 hashed. Operators are created as: id = "OP-001", is_active = True. These operators can then authenticate via the OTP flow.
6. Return Seed Summary
JSON response with counts of all seeded entities
Response: { "status": "seeded", "stops": 95, "routes": 25, "trips": 350, "operators": 2 }

Additionally, /api/v1/admin/db-status provides a health check:
GET /api/v1/admin/db-status
Returns: { "status": "connected", "routes": 25, "stops": 95, "trips": 350 }
